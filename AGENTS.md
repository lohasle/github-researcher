【GitHub 数据获取与可靠性规则】
为了提高任务稳定性和数据获取质量，请遵守以下规则：

1. 获取 GitHub 数据时，若环境中可用 `gh`（GitHub CLI），优先使用 `gh`
   - 优先用于获取仓库信息、README、issue、PR、release、workflow、文件内容、目录结构等 GitHub 数据
   - 若 `gh` 不可用，再退回使用 git、本地仓库、网页抓取或其他可行方式
   - 若同时存在多种方式，优先选择更稳定、结构化、更适合自动化处理的方式

2. 对以下高风险步骤，默认启用有限重试策略：
   - GitHub 数据抓取
   - git pull
   - git push
   - GitHub Pages 可用性检查
   - 关键页面链接检查

3. 网络异常、GitHub 服务波动、短时 DNS / TLS / 超时错误，不应在第一次失败后立刻放弃
   - 对这类错误，应优先视为暂时性错误，并进行有限次重试

4. 如果重试后仍失败，必须在输出中明确说明：
   - 失败发生在哪一步
   - 已经尝试了几次
   - 当前仓库有哪些本地修改尚未成功推送
   - 对本次研究结果是否产生影响
   - 建议如何补救

5. 当 GitHub 数据抓取失败时：
   - 先尝试重试
   - 若仍失败，可尽量基于已同步的本地仓库数据完成研究
   - 但必须明确标注哪些信息来自最新抓取，哪些信息来自已有本地数据或历史状态

6. 当 git push 失败时：
   - 先检查是否为网络抖动或远端短时不可达
   - 重试后若仍失败，必须保留本地修改结果，不得丢失
   - 输出待推送文件列表与失败原因

7. 可靠性优先于花哨实现
   优先保证：
   - 数据尽量抓到
   - 研究尽量完成
   - 仓库尽量更新
   - 页面尽量发布
   - 失败时结果不丢
