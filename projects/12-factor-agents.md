# 12-Factor Agents

## 一句话定位
构建足够好到可以交给专业用户使用的 LLM 驱动软件的 12 条工程原则，Agent 领域的「12-Factor App」。

## 它解决的问题
Agent 开发目前处于「每个人都在摸索」的阶段。大量 Agent 项目是实验性的，缺乏工程原则指导。团队在构建 Agent 时不知道什么才是「好的」做法。

## 为什么值得关注（2026-05-22）

1. **里程碑意义**：Agent 领域首次出现系统性的工程原则定义
2. **实战驱动**：不是学术理论，而是来自 humanlayer 团队的实践总结
3. **可操作性强**：12 条原则可以直接转化为团队开发规范
4. **产业信号**：当领域开始定义「最佳实践」时，说明正在从实验期进入工程期

## 关键技术亮点
- 12 条原则覆盖 Agent 软件的核心维度
- TypeScript 参考实现
- 强调「足够好」而非「完美」— 实用主义

## 架构启发
- Agent 软件应该有明确的工程标准
- 从 12-Factor App 到 12-Factor Agents — 模式复现
- 原则先行，实现跟进

## 定位判断
**基础设施候选** — 虽然不是代码级基础设施，但作为工程原则，它是 Agent 架构的「基础设施」。

## 风险/局限/泡沫点
- 原则的普适性有待更多实践验证
- 可能过于 TypeScript/JS 生态偏重
- 12 条原则可能不覆盖所有场景（如多 Agent 协作）
- 存在「定义权争夺」的风险

## 与同类项目的关系
- **multica** (30.6K⭐)：实践层面的 Agent 平台，互补
- **superpowers** (201K⭐)：技能框架，从工具层支撑
- **CLI-Anything** (39K⭐)：Agent-Native 范式的具体实现

## 是否值得持续跟踪
**是** — 强烈建议持续跟踪，可作为内部 Agent 项目评估框架。

## 后续观察点
- 社区对 12 条原则的反馈和修改
- 是否有企业基于此做内部标准
- 是否出现「认证」或「合规」机制
- 其他厂商是否有类似原则提出
