# React Doctor

## 一句话定位
你的 Agent 写了烂 React，这个工具来捕获它。AI 编码质量守门员。

## 它解决的问题
AI Coding Agent 生成的 React 代码表面上能运行，但常常存在反模式、性能陷阱和可维护性问题。React Doctor 专门检测 AI 生成的 React 代码中的常见问题。

目标用户：使用 AI 编写 React 的开发者、技术负责人、Code Reviewer。

## 为什么值得关注（2026-05-19）

10.2K⭐，GitHub Trending 本周。millionco 出品。「AI 写的代码质量」是当前 AI 编程社区的热点话题。专用工具检测 AI 编码问题的方向正确。

## 热度来源判断
热度 70% 真实需求。AI 编码质量确实存在问题，开发者有感知。但作为单一框架（React）的质量工具，天花板有限。

## 关键技术亮点
1. **AI 编码反模式检测**：专门针对 AI 生成代码的常见问题（过度抽象、不必要的 useEffect 等）
2. **TypeScript 实现**：与 React 生态技术栈一致
3. **CI/CD 集成友好**：可作为 PR 检查的一部分

## 架构启发
- AI 编码质量守门是 AI 编程工具链的新品类
- 框架专用（vs 通用）的质量工具在初期更有价值

## 定位判断
AI 编程质量工具链的细分工具。类似 ESLint 之于代码规范，React Doctor 之于 AI 编码质量。

## 风险 / 局限 / 泡沫点
1. **React-only**：只覆盖 React 生态，天花板明显
2. **规则维护**：AI 编码模式持续进化，检测规则需要持续更新
3. **与 Linter 重叠**：部分功能与 ESLint 规则重叠

## 与同类项目的关系
- **vs ESLint**：ESLint 是通用 linter，React Doctor 专注 AI 编码反模式
- **vs SonarQube**：SonarQube 更全面但更重，React Doctor 更轻量专注

## 是否值得持续跟踪
**低优先级。** 方向正确但单一框架工具天花板有限。

## 后续观察点
1. 是否扩展到 Vue/Svelte 等其他框架
2. AI 编码模式是否会自我改进到不需要专用检测
3. 是否形成 AI 编码质量标准
